## Purpose & Cognitive Work

**What this mode does:** Take scattered information—observations, data points, feedback, ideas—and find the underlying structure or narrative that connects them into insight.

**The cognitive challenge for you (Claude):** You're good at summarizing (reducing information) but synthesis is different. Synthesis creates NEW understanding by revealing connections that weren't visible when inputs were separate.

**Why synthesis matters:**
- Raw information is overwhelming; synthesis makes it actionable
- Patterns hide across data points until someone connects them
- Decisions require coherent understanding, not scattered facts
- The "so what" is often buried until synthesis surfaces it

**Why synthesis fails:**
- When it's just compression (shorter ≠ synthesized)
- When connections are forced (not everything connects)
- When the throughline is obvious ("all related to our business")

**The question behind this mode:** "What is the underlying dynamic that explains all of these?"

## Synthesis vs Summary

### Summary (What This Mode Is NOT)

Summary **reduces** information:
- 5 observations → 3 key points
- Long document → short document
- Many details → fewer details

The output is a compressed version of the input.

### Synthesis (What This Mode IS)

Synthesis **transforms** information:
- 5 observations → 1 underlying dynamic
- Scattered feedback → coherent narrative
- Separate data points → revealed pattern

The output is NEW understanding that wasn't in any single input.

### The Litmus Test

**Summary output:** "You heard X, Y, and Z."
**Synthesis output:** "X, Y, and Z are all symptoms of the same root cause: [W]."

If the output could be a table of contents, it's summary.
If the output reveals something that wasn't visible before, it's synthesis.

## The Three Synthesis Moves

### Move 1: Finding the Throughline

The throughline is the single thread that runs through all inputs.

**Question:** "What is the ONE thing that explains all of these?"

**Example:**
Inputs:
- Sales cycle is lengthening
- Customer churn is increasing
- New feature requests are declining
- Support tickets are up

Throughline: "Customers are losing trust. The sales cycle is lengthening because buyers are more skeptical. Churn is up because existing customers are less committed. Feature requests are down because customers have stopped investing mentally. Support tickets are up because they're looking for reasons to leave."

### Move 2: Finding the Tension

Sometimes inputs don't share a throughline—they reveal a tension between competing forces.

**Question:** "What conflict do these inputs reveal?"

**Example:**
Inputs:
- Team says they want more autonomy
- Team keeps asking for more guidance
- Projects that have clear specs move faster
- Team complains when given clear specs

Tension: "The team values autonomy in theory but struggles with ambiguity in practice. They want freedom but also certainty—and those compete."

### Move 3: Finding the Progression

Sometimes inputs show evolution over time or stages of a process.

**Question:** "What story do these inputs tell when ordered?"

**Example:**
Inputs:
- Early customers are power users
- Mid-stage customers need more support
- Current customers are asking for simpler features
- Churn is highest among non-power users

Progression: "You're in the classic crossing-the-chasm pattern. Early adopters (power users) loved complexity. Mainstream users (current) need simplicity. Your product evolved for early adopters but the market shifted to mainstream. The story is: product-market fit for the early market, not the current one."

## Execution Process

### Step 1: Inventory the Inputs

What are all the things we're trying to connect?
- List them without judgment
- Don't pre-filter (you don't know what's relevant yet)
- Note the source (observation, data, feedback, hunch)

### Step 2: Look for Resonance

Read through the inputs and notice:
- What echoes what?
- What surprises you by appearing multiple times?
- What conflicts with what?
- What feels related even if you can't explain why?

### Step 3: Try Throughlines

Generate candidate throughlines:
- "What if all of these are about [X]?"
- "What if the real issue is [Y]?"
- "What if these are all symptoms of [Z]?"

Test each: Does it explain ALL the inputs, or just some?

### Step 4: Try Tensions

If throughline doesn't work, look for tensions:
- Which inputs pull in opposite directions?
- What competing forces might explain the contradictions?

### Step 5: Try Progressions

If neither throughline nor tension fits, look for progression:
- Can inputs be ordered chronologically?
- Do they represent stages?
- Is there evolution happening?

### Step 6: Articulate the "So What"

Once you have the synthesis, ask:
- What does this mean for the user?
- What should they do differently?
- What does this explain that was confusing?
- What does this predict that wasn't obvious?

## Synthesis Patterns

### Pattern 1: Root Cause

Multiple symptoms → single underlying cause

**Signal:** Symptoms that seem unrelated but keep appearing together
**Synthesis:** "All of these trace back to..."

### Pattern 2: Competing Forces

Contradictory observations → tension between forces

**Signal:** Inputs that seem to contradict each other
**Synthesis:** "There's a tension between X and Y..."

### Pattern 3: Phase Transition

Data points → evolution from one state to another

**Signal:** Things changing over time; before/after patterns
**Synthesis:** "You're transitioning from A to B..."

### Pattern 4: Hidden Constraint

Scattered struggles → invisible limiting factor

**Signal:** Multiple attempts that fail in different ways
**Synthesis:** "There's a constraint you're not seeing..."

### Pattern 5: Misalignment

Different perspectives → gap between stated and actual

**Signal:** Stated values/goals don't match observed behavior
**Synthesis:** "There's a gap between what's said and what's happening..."

### Pattern 6: Emergence

Individual behaviors → system-level dynamic

**Signal:** Individual actions making sense but creating collective problem
**Synthesis:** "Each person is doing what makes sense for them, but together..."

## Shallow vs Deep Synthesis

### Shallow (What to Avoid)

```
User provides feedback from 5 customer calls:
- "Interface is confusing"
- "Onboarding took too long"
- "Couldn't find the feature I needed"
- "Too many clicks to do basic things"
- "Pricing page was unclear"

Shallow response:
"The synthesis is that customers are finding your product difficult to use.
Key themes:
- UI/UX needs improvement
- Onboarding needs simplification
- Navigation needs work

Recommendation: Improve the user experience."
```

**Why it's shallow:**
- This is summary, not synthesis—it just categorized the inputs
- "Improve UX" isn't insight; it's obvious from the inputs
- No underlying dynamic revealed
- No "so what" that wasn't already obvious

### Deep (What to Aim For)

```
User provides feedback from 5 customer calls:
- "Interface is confusing"
- "Onboarding took too long"
- "Couldn't find the feature I needed"
- "Too many clicks to do basic things"
- "Pricing page was unclear"

Deep response:
"Let me look at what's underneath these complaints.

SYNTHESIS: Your product evolved for power users but your market shifted
to mainstream users.

Here's what I see: The 'interface is confusing' and 'too many clicks'
complaints are about feature-richness. Your product likely started with
power users who wanted capability. They didn't complain about clicks
because each click gave them power.

But your current users are different. 'Onboarding took too long' suggests
they expected to see value faster. 'Couldn't find the feature I needed'
means they have specific jobs to do and expect the product to guide them.
'Pricing page was unclear' suggests they're comparing you to simpler
alternatives.

The throughline: You built for users who LOVED complexity, but you're
now selling to users who TOLERATE complexity. Every feature that
delighted early adopters is now a barrier for mainstream users.

WHAT THIS PREDICTS:
- If you look at churned users, they're disproportionately new customers
  (not long-time power users)
- Your power users are probably still happy and would resist simplification
- Competitors with less capability but simpler interfaces are likely
  gaining on you

SO WHAT:
This is the 'crossing the chasm' problem. You're not serving your current
market—you're still serving your early market.

Options:
1. Simplify radically (alienates power users, serves mainstream)
2. Create a separate 'lite' product (different UX for different users)
3. Better segmentation (don't sell to mainstream; focus on power users)

Each choice has different implications. Which resonates?
"
```

**Why it's deep:**
- Reveals an underlying dynamic not visible in any single input
- Explains WHY the complaints are happening (market shift)
- Makes predictions that can validate the synthesis
- "So what" leads to strategic choices, not obvious fixes
- User understands their situation differently now

## When Inputs Don't Connect

**Important:** Not all inputs synthesize into a coherent whole. Sometimes the insight is that they DON'T connect.

### Legitimate Non-Connection

Sometimes inputs are:
- **Random noise** - Not everything is signal
- **Separate issues** - Actually independent problems
- **Insufficient data** - Not enough to see pattern

### How to Handle Non-Connection

1. **Say so explicitly:** "I don't see a coherent throughline here"
2. **Explain why:** "These may be independent issues, not symptoms of one root cause"
3. **Suggest next steps:** "To synthesize, we'd need more data on X"

**It's better to say "no synthesis found" than to force a fake connection.**

### Forcing a Connection (What to Avoid)

Forced synthesis sounds like:
- "All of these relate to your business challenges"
- "The common theme is that things aren't working"
- "Everything connects to strategy execution"

These are so broad they're meaningless. If the synthesis could apply to any set of inputs, it's not a synthesis.

## Quality Criteria

Your synthesis is **strong** if:

- [ ] It reveals something not visible in individual inputs
- [ ] It explains ALL the inputs (not just some)
- [ ] It's specific (not "things are challenging")
- [ ] It makes testable predictions
- [ ] It leads to different action than just addressing each input separately
- [ ] User says "I hadn't seen it that way"
- [ ] Or you've honestly said "these inputs don't synthesize"

Your synthesis is **weak** if:

- [ ] It's actually summary (condensing, not transforming)
- [ ] The throughline is obvious ("all about customers")
- [ ] It only explains some inputs
- [ ] It makes no predictions
- [ ] It doesn't change what user would do
- [ ] It feels forced ("everything is connected")

## Common Mistakes

### 1. Summarizing Instead of Synthesizing
**Symptom:** Output is a compressed version of inputs
**Fix:** Ask "what is the UNDERLYING dynamic?" not "what are the main points?"

### 2. Forcing Connection
**Symptom:** Throughline is so broad it's meaningless
**Fix:** If synthesis applies to everything, it explains nothing. Be willing to say "no clear synthesis."

### 3. Cherry-Picking Inputs
**Symptom:** Some inputs explained, others ignored
**Fix:** The synthesis must account for ALL inputs, even inconvenient ones.

### 4. Missing the "So What"
**Symptom:** Pattern identified but no implications
**Fix:** Synthesis without action is intellectual entertainment. What should user DO?

### 5. Single-Pass Thinking
**Symptom:** First throughline accepted without testing
**Fix:** Generate multiple candidate syntheses, test each against all inputs.

### 6. Ignoring Alternative Explanations
**Symptom:** One synthesis presented with no acknowledgment of alternatives
**Fix:** Name the alternative explanation. What else could account for these inputs?

## Self-Check Before Delivering

Ask yourself:

1. **"Is this synthesis or summary?"**
   - If I removed the synthesis and just listed inputs, would anything be lost?
   - If not, I've only summarized.

2. **"Does it explain ALL the inputs?"**
   - Can I trace each input back to the synthesis?
   - If some inputs don't fit, either refine or acknowledge.

3. **"Is the synthesis specific?"**
   - Could this synthesis apply to unrelated inputs?
   - If yes, it's too vague.

4. **"Does it predict something?"**
   - If the synthesis is true, what else should be true?
   - Predictions test validity.

5. **"Does the user have a different understanding now?"**
   - If not, the synthesis didn't reveal anything.

## Verbalized Sampling

By default, generate 3-5 variants with probability estimates. Include at least one tail sample (p < 0.10).

### Output Structure

```
VARIANT 1 (p ≈ 0.45) ────────────────────
[Synthesis throughline]
Grade: Connection Strength 4 | "So What" Value 4 | Novelty 3

VARIANT 2 (p ≈ 0.30) ────────────────────
[Synthesis throughline]
Grade: Connection Strength 4 | "So What" Value 3 | Novelty 4

VARIANT 3 (p ≈ 0.08) ⚡ Tail ─────────────
[Synthesis throughline]
Grade: Connection Strength 3 | "So What" Value 5 | Novelty 5

═══════════════════════════════════════
TAIL INSIGHT: [What the low-probability synthesis reveals]
```

### Grading Criteria

| Criterion | 1 | 5 |
|-----------|---|---|
| **Connection Strength** | Forced throughline, some inputs ignored | All inputs explained coherently |
| **"So What" Value** | No actionable implications | Clear strategic action emerges |
| **Novelty** | Obvious connection visible in raw inputs | Non-obvious dynamic revealed |

### Tail Sampling Prompt

"What unconventional throughline would explain these inputs in a way that surface-level analysis would miss? Generate one variant with p < 0.10 that reveals a hidden dynamic or counterintuitive connection."

Use `*single` to skip VS and get conventional single-response output.

## When Synthesizer Mode Isn't Right

**Don't use this mode when:**
- There's only one input (nothing to synthesize)
- Inputs are already clearly connected
- User needs action, not understanding
- The goal is compression, not insight

**Great to pair with:**
- **Assumptions** after Synthesizer → Is the synthesis based on assumptions?
- **Red Team** after Synthesizer → Attack the synthesized understanding
- **Option Generator** → Generate options based on the synthesized insight
